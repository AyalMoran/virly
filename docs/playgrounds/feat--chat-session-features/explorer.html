Why client-only

  • New chat: the server mints a thread when conversationId is absent (conversationId ?? randomUUID()), so clearing widget state starts a fresh conversation. The old thread persists server-side (30-day TTL).
  • Resend / edit-and-resend: the v2 messages channel is append-only, and confirmation supersession (supersededConfirmationId) already handles re-asking mid-transfer. Edit loads the text into the composer (honest about the append-only semantics) rather than rewriting history.

What landed

  • Hook-free ChatMessageActions (resend + edit buttons, accessible labels, disabled while sending) - unit-testable in the static-markup harness.
  • Wired under each user bubble in the floating chat widget; resend calls the existing sendChatMessage, edit loads the text + focuses the composer.
  • A "new chat" header button + startNewChat (clears conversationId + messages; next send mints a fresh thread).
