The "startup throttle" delayed every API call

Todoist 6h2Rpwm7rQG9XH76: "the throttle at the start affects every api call; every page takes seconds to load." It did - a dev latency simulator was left switched on, and it was never startup-only.

bug: 6s on every request server-only + production kill-switch PR #32

Root cause

server/.env carried a stale VIRLY_THROTTLE_MS=6000, and the inline middleware in server/src/app.ts applied it to every request - registered before all /api/* routes, with no production guard. A trivial GET /api/health (no DB, no auth) measured 6.01 s. The task called it a "startup" throttle; the implementation was always global. 

// BEFORE - server/src/app.ts
const throttleMs = Number(process.env.VIRLY_THROTTLE_MS) || 0;
if (throttleMs > 0) {
  app.use((_req, _res, next) => { setTimeout(next, throttleMs); });  // every request, no guard
}

The fix

  • server/src/middleware/devThrottle.ts (new): resolveDevThrottleMs centralizes the env read and returns 0 when NODE_ENV=production - a leftover value can never slow real traffic. warnDevThrottleActive logs a loud boot warning while active. Unit-tested (5 cases).
  • app.ts routes through the guarded module, in the same middleware position.
  • Deleted the live VIRLY_THROTTLE_MS=6000 from local server/.env; documented the real semantics in server/.env.example and docs/configuration.md.
// AFTER
const throttleMs = resolveDevThrottleMs({
  VIRLY_THROTTLE_MS: process.env.VIRLY_THROTTLE_MS,
  NODE_ENV: process.env.NODE_ENV
});
if (throttleMs > 0) {
  warnDevThrottleActive(throttleMs);
  app.use(devThrottleMiddleware(throttleMs));
}
